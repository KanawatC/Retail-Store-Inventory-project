# Retail-Store-Inventory-project

Data cleaning and exploratory analysis of a two-year retail inventory dataset:
73,100 daily records across 5 stores, 20 product slots, 5 categories and 4
regions (2022-01-01 to 2024-01-01).

## Project structure

```
data/raw/retail_store_inventory.csv        source data (73,100 rows, 15 columns)
data/cleaned/retail_store_inventory_cleaned.csv   cleaned dataset (27 columns)
notebooks/01_profile.py                    raw-data profiling (types, nulls, dupes, ranges)
notebooks/02_clean.py                      cleaning: types, invalid values, derived fields
notebooks/03_eda.py                        EDA aggregations -> outputs/eda_summary.json
notebooks/04_build_report.py               SVG chart generation -> outputs/charts_cache.json
notebooks/05_assemble_html.py              assembles the HTML report
outputs/eda_summary.json                   aggregated EDA stats
outputs/charts_cache.json                  generated chart markup
```

## Data cleaning

- **Missing values:** none found across all 15 source columns.
- **Duplicates:** none found (exact rows, and `date` + `store_id` + `product_id`
  + `category` keys). The panel is complete: 731 days x 5 stores x 20 product
  slots = 73,100 rows exactly.
- **Invalid values:** 673 rows (0.92%) had a negative `Demand Forecast` (as low
  as -9.99), which isn't possible for a demand quantity. Clipped to 0 and
  flagged in `demand_forecast_was_negative` rather than dropped.
- **Types:** `Date` parsed to datetime; `Store ID`, `Product ID`, `Category`,
  `Region`, `Weather Condition`, `Seasonality` cast to categorical;
  `Holiday/Promotion` cast to boolean; columns renamed to `snake_case`.
- **Derived fields:** `revenue`, `forecast_error`, `abs_forecast_error`,
  `stockout_flag`, `sell_through_rate`, `price_vs_competitor`, plus calendar
  fields (`year`, `month`, `weekday`, `is_weekend`).

## Data analysis (EDA)

Key findings from the cleaned dataset:

- Revenue and units sold are essentially flat across category, region, store,
  season, weekday, weather and discount tier â€” none of these move average
  units sold by more than ~2%.
- Stock is rarely a binding constraint: 0.50% stockout rate and ~49.8% average
  sell-through, i.e. stores carry roughly double the inventory they sell
  through day to day.
- `price` and `competitor_pricing` move in lockstep (r = 0.994), and the
  store's own price sits above the competitor's on only 49.9% of rows â€”
  consistent with competitor pricing being generated as a function of the
  store's own price rather than observed independently.
- `product_id` is **not** a stable product key: each of the 20 IDs cycles
  through all 5 categories within a store over the two years, so it behaves
  as a recycled slot code rather than a persistent SKU.

### Forecast accuracy â€” conclusion

The dataset ships with a pre-existing `Demand Forecast` column alongside
actual `Units Sold`. No forecasting model was built as part of this analysis â€”
the "forecast" was evaluated as-is by comparing it to actuals
(`forecast_error = units_sold - demand_forecast`, aggregated to standard
error metrics):

| Metric | Value |
|---|---|
| Mean bias | -5.06 units/day |
| MAE | 8.30 units |
| RMSE | 10.00 units |
| MAPE | 23.3% |
| Correlation (forecast vs. actual) | r = 0.997 |
| Rows with invalid (negative) raw forecast | 673 (0.92%) |

**Conclusion:** the forecast is directionally excellent â€” it tracks actual
demand almost perfectly (r = 0.997) â€” but it is not *unbiased*. It
over-predicts actual units sold by about 5 units per store-day, consistently,
across every category (not a handful of outliers dragging the average). Because
the error is a systematic offset rather than random noise, it is easy to
correct: subtracting a fixed bias term (or re-centering the model) would cut
the MAE roughly in half without needing a new model. Left uncorrected, this
bias would push replenishment decisions built on the raw forecast toward
consistent over-ordering. The 0.92% of rows with a negative raw forecast
point to the same issue at the model-output level and should be clipped or
re-fit at the source, not just patched downstream. This is an audit of an
existing forecast column, not a forecasting model built during this analysis.
