# Retail-Store-Inventory-project

Data cleaning and exploratory analysis of a two-year retail inventory dataset:
73,100 daily records across 5 stores, 20 product slots, 5 categories and 4
regions (2022-01-01 to 2024-01-01).

## Project structure

```
data/raw/retail_store_inventory.csv        source data (73,100 rows, 15 columns)
data/cleaned/retail_store_inventory_cleaned.csv   cleaned dataset (27 columns)
notebooks/01_profile.py                    raw-data profiling (types, nulls, dupes, ranges)
notebooks/02_clean.py                      cleaning: types, invalid values, derived fields
notebooks/03_eda.py                        EDA aggregations -> outputs/eda_summary.json
notebooks/04_build_report.py               SVG chart generation -> outputs/charts_cache.json
notebooks/05_assemble_html.py              assembles the HTML report
outputs/eda_summary.json                   aggregated EDA stats
outputs/charts_cache.json                  generated chart markup
```

## Data cleaning

- **Missing values:** none found across all 15 source columns.
- **Duplicates:** none found (exact rows, and `date` + `store_id` + `product_id`
  + `category` keys). The panel is complete: 731 days x 5 stores x 20 product
  slots = 73,100 rows exactly.
- **Invalid values:** 673 rows (0.92%) had a negative `Demand Forecast` (as low
  as -9.99), which isn't possible for a demand quantity. Clipped to 0 and
  flagged in `demand_forecast_was_negative` rather than dropped.
- **Types:** `Date` parsed to datetime; `Store ID`, `Product ID`, `Category`,
  `Region`, `Weather Condition`, `Seasonality` cast to categorical;
  `Holiday/Promotion` cast to boolean; columns renamed to `snake_case`.
- **Derived fields:** `revenue`, `forecast_error`, `abs_forecast_error`,
  `stockout_flag`, `sell_through_rate`, `price_vs_competitor`, plus calendar
  fields (`year`, `month`, `weekday`, `is_weekend`).

## Data analysis (EDA)

Key findings from the cleaned dataset:

- Revenue and units sold are essentially flat across category, region, store,
  season, weekday, weather and discount tier — none of these move average
  units sold by more than ~2%.
- Stock is rarely a binding constraint: 0.50% stockout rate and ~49.8% average
  sell-through, i.e. stores carry roughly double the inventory they sell
  through day to day.
- `price` and `competitor_pricing` move in lockstep (r = 0.994), and the
  store's own price sits above the competitor's on only 49.9% of rows —
  consistent with competitor pricing being generated as a function of the
  store's own price rather than observed independently.
- `product_id` is **not** a stable product key: each of the 20 IDs cycles
  through all 5 categories within a store over the two years, so it behaves
  as a recycled slot code rather than a persistent SKU.

### Forecast accuracy — conclusion

The dataset ships with a pre-existing `Demand Forecast` column alongside
actual `Units Sold`. No forecasting model was built as part of this analysis —
the "forecast" was evaluated as-is by comparing it to actuals
(`forecast_error = units_sold - demand_forecast`, aggregated to standard
error metrics):

| Metric | Value |
|---|---|
| Mean bias | -5.06 units/day |
| MAE | 8.30 units |
| RMSE | 10.00 units |
| MAPE | 23.3% |
| Correlation (forecast vs. actual) | r = 0.997 |
| Rows with invalid (negative) raw forecast | 673 (0.92%) |

**Conclusion:** the forecast is directionally excellent — it tracks actual
demand almost perfectly (r = 0.997) — but it is not *unbiased*. It
over-predicts actual units sold by about 5 units per store-day, consistently,
across every category (not a handful of outliers dragging the average). Because
the error is a systematic offset rather than random noise, it is easy to
correct: subtracting a fixed bias term (or re-centering the model) would cut
the MAE roughly in half without needing a new model. Left uncorrected, this
bias would push replenishment decisions built on the raw forecast toward
consistent over-ordering. The 0.92% of rows with a negative raw forecast
point to the same issue at the model-output level and should be clipped or
re-fit at the source, not just patched downstream. This is an audit of an
existing forecast column, not a forecasting model built during this analysis.

## Challenge 1: Time series demand forecasting

**Task:** predict daily product demand across stores, and test whether an
LSTM can outperform a classical method like ARIMA.

> **Redo note:** the first pass forecast one flat "total demand" number for
> the whole chain. That's not actionable for stocking — a warehouse needs to
> know how much *Furniture* vs. *Groceries* to hold, not one company-wide
> figure. This version forecasts demand **per product category**, summed
> across stores, and turns store back into an even split at the end (see
> Step 1). The pipeline below supersedes the original.

```
forecasting/01_hypothesis_granularity.py   ANOVA: does demand depend on store/product/category/region? (kept from v1)
forecasting/10_category_series.py          daily demand per category, summed across all 5 stores
forecasting/11_diagnostics_category.py     feature significance + ADF/trend/ACF, re-run per category
forecasting/12_baselines_category.py       mean / naive / seasonal-naive baselines, per category
forecasting/13_arima_category.py           5 independent grid-searched ARIMA fits
forecasting/14_lstm_global.py              1 global multi-series LSTM (category as input), blind forecast
forecasting/15_compare_and_stock_plan.py   consolidated metrics + per-store stock allocation
forecasting/16_build_report.py             chart generation
forecasting/17_assemble_html.py            assembles the HTML report
```

**Step 1 — granularity, corrected.** The original ANOVA (`01_hypothesis_granularity.py`)
showed store, product, category and region all move demand by a practically
negligible amount at the row level (eta-squared < 0.0003) — but that test
answers "does the *average level* differ," not "do I need separate numbers to
run a warehouse." Furniture and Groceries are physically different goods
regardless of whether their average daily volumes are statistically close, so
**category is the forecast unit**, not the full aggregate. Store, by contrast,
really can be dropped: re-testing store's effect *within each category
separately* confirms it stays negligible everywhere (eta-squared 0.0000–0.0007
across all 5 categories) — an even 1/5 split across stores is a statistically
defensible stocking rule per category, not just on average.

**Step 2 — features and diagnostics, re-run per category.** Price, discount,
promotion, weather, season and weekday were re-tested against each category's
own daily demand (5 categories x 8 variables = 40 tests). Zero survived
p<0.05 *and* effect>=0.02 — the univariate decision from the first pass holds
per category too. All five series are stationary; Groceries is the only one
with a statistically real (if small) downward trend (p=0.009).

**Step 3 — models**, evaluated on a 90-day blind holdout per category (5
independent ARIMA fits vs. 1 global multi-series LSTM — LSTM(32) + a 5-dim
category one-hot, trained jointly across all 5 series):

| Category | Mean baseline MAE | ARIMA MAE | LSTM MAE | Best on test |
|---|---|---|---|---|
| Clothing | 595.5 | ARIMA(0,0,1): 595.1 | 665.9 | ARIMA (~tied with mean) |
| Electronics | 606.5 | ARIMA(0,0,2): 605.9 | **589.4** | **LSTM** |
| Furniture | **617.7** | 618.6 | 636.0 | mean baseline |
| Groceries | **563.5** | ARIMA(0,1,0): 822.6 | 627.0 | mean baseline |
| Toys | **562.5** | 564.8 | 581.0 | mean baseline |
| **Average** | **589.1** | **641.4** | **619.9** | — |

Groceries is the cautionary tale: its order search picked a differenced
random-walk model to chase the category's small real trend — which then
didn't continue into the test window, making it ARIMA's worst result by far.

**Step 4 — the practical output.** Chain-wide daily demand (historical mean —
the strongest single estimator across the comparison) split evenly across the
5 stores:

| Category | Chain daily demand | Per store, per day |
|---|---|---|
| Clothing | 2,751 | 550 |
| Electronics | 2,661 | 532 |
| Furniture | 2,746 | 549 |
| Groceries | 2,752 | 550 |
| Toys | 2,728 | 546 |

### Conclusion

**LSTM still beats ARIMA, and by a clearer margin than the first pass**
(average MAE 619.9 vs. 641.4, +3.4%, up from +3.6% in the single flat-series
version — comparable, but this time visibly *why*: a single global model
trained across all 5 categories isn't exposed to the per-series overfitting
that sank ARIMA's Groceries fit). **Neither model beats the historical mean by
much**, confirming at finer grain what Steps 1-2 already established: no other
variable in the dataset carries usable signal, so a category's own recent
average remains the strongest available predictor. The real fix in this redo
isn't the model — it's the target: forecasting five actionable per-category
numbers (and a defensible per-store split) instead of one number nobody could
act on. Recommendation: use the mean-based stock plan above as the day-to-day
default, and keep the global LSTM as the forecasting method of choice if this
were extended to more product types — it scales as one model rather than one
per series, and it's the more robust of the two learned models. Treat
per-category ARIMA with caution on any series with a borderline-significant
trend.
